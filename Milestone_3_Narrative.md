# CS 499 Milestone Three Narrative

## Artifact Description

The artifact selected for this enhancement is the Travlr Getaways web application originally developed in CS 465: Full Stack Development. The original project was built as a MEAN-stack application using MongoDB, Express, Angular, and Node.js. It included customer-facing travel content, REST API functionality, database integration, authentication support, and an Angular administrative interface for managing trip data. In Milestone Two, I began migrating this artifact toward a React, Spring Boot, and PostgreSQL architecture by creating a working vertical slice with a React client and Spring Boot API.

Milestone Three continues that migration by focusing on the algorithms and data structures enhancement. The original Travlr application could retrieve and display trip data, but its trip processing logic was limited. The application was not originally designed around a flexible query layer or derived catalog metadata. For this milestone, I enhanced the Spring Boot API so trip records can be processed through structured query behavior rather than simply returned as a static collection. This functionality is currently implemented against the in-memory repository, but it is intentionally organized behind the service and repository layers so the same behavior can be moved into the PostgreSQL database layer in Milestone Four.

## Justification for ePortfolio Inclusion

I selected this artifact for my ePortfolio because it shows growth beyond the original guided course project. The Travlr application already had a realistic business purpose and multiple application layers, which made it a strong foundation for capstone enhancement. However, the original version left room for improvement in how trip data was processed after retrieval. Instead of selecting a small standalone algorithm assignment, I wanted to show that algorithms and data structures also matter in application development, especially when a system needs to organize and return data according to user-selected criteria.

This artifact is appropriate for the algorithms and data structures category because the Milestone Three enhancement expands the trip API from basic retrieval into a structured query system. Building on the typed model and layered backend created in Milestone Two, I added broader query behavior through TripSearchCriteria, service-layer processing, comparator-based ordering, bounded result handling, and derived catalog metadata through TripSummary. These changes demonstrate that the application is no longer only retrieving trip records, it is applying algorithmic logic to organize and return results based on caller-selected criteria.

The data structure work was also strengthened in this milestone. The in-memory repository now maintains trip records in a List<Trip> for ordered collection traversal while also building a Map<String, Trip> keyed by normalized trip code for efficient single-record lookup. This improves on a simple linear search approach and reinforces the idea that trip codes are stable identifiers. The current implementation uses in-memory structures because the PostgreSQL layer is planned for Milestone Four, but the repository boundary keeps that transition clear. Later, the same lookup and query behavior can be supported through database indexes, constraints, and repository queries.

## Enhancement Summary

The main improvement made during this milestone was expanding the Spring Boot trip API from basic read-only retrieval into a structured query layer. The repository and service structure created in the previous milestone made this possible because trip data access was already separated from controller logic. In Milestone Three, I built on that structure by strengthening the in-memory repository with a map-backed trip-code lookup and by expanding the service layer so it can process trip collections according to optional query criteria.

The TripService class became the center of the algorithms and data structures enhancement. It retrieves trip records from the repository and applies the query pipeline for keyword matching, range filtering, approved ordering, sort direction, and bounded result pages. The service also includes a summary operation that derives catalog metadata such as the total number of trips, distinct resort values, minimum and maximum price, and minimum and maximum trip duration. This metadata can support future frontend filter controls without requiring the client to duplicate collection-scanning logic.

The API now exposes read-only endpoints for retrieving trips, retrieving one trip by code, and retrieving trip summary metadata. Query parameters can be used to control the returned results. The current implementation still uses in-memory data because PostgreSQL will be added in Milestone Four, but the query behavior is intentionally organized behind the service and repository layers. This keeps the current enhancement useful while preserving a clean path for moving the added functionality into database-backed queries, indexes, and constraints later.

I also strengthened the milestone by adding automated tests. The repository tests verify that the current in-memory repository exposes application trip data and supports case-insensitive trip-code lookup. The service tests use controlled test data to verify the main query behavior independently from the application’s seed data. I also added defensive-input tests to show that unexpected user-controlled query values are handled safely, including unsupported sort fields, invalid sort directions, negative page numbers, oversized page sizes, blank trip codes, and unknown trip codes.

## Design Improvements, Learning, and Challenges

The enhancement helped me think more carefully about how algorithms and data structures appear in normal application development. The important decision was not to add a complicated algorithm for its own sake, but to improve the way the application processes trip data. A list fits ordered collection traversal and result processing, while a map fits direct lookup by trip code. Even though the current seed data set is small, choosing structures based on access patterns helped me connect the milestone work to the kind of indexing and query behavior that will be needed once PostgreSQL is added.

I also learned more about the order of operations in a query pipeline. Filtering needs to happen before sorting so the application does not sort records that will be excluded. Sorting needs to happen before pagination so the requested page comes from the final ordered result set. Pagination also needs defensive bounds so invalid or oversized requests do not create unexpected behavior. These choices are simple individually, but together they make the query process more predictable and easier to test.

One challenge was making the algorithms and data structures enhancement meaningful in a project that was not originally algorithm-heavy. The original Travlr application was primarily a full-stack development artifact, not an algorithms assignment. To address this, I focused on places where algorithmic thinking naturally fit the application: collection traversal, comparator selection, map-backed lookup, query ordering, bounded result handling, and derived catalog metadata. This strengthened the project without adding an artificial algorithm that did not match the purpose of the application.

Another challenge was balancing the current milestone with the future database milestone. My Module One plan included search, filtering, sorting, and ranking, with trade-offs between database-side filtering and service-layer ranking logic. In this milestone, I completed the practical query behavior that fits the current in-memory implementation and added defensive input handling around it. I chose not to fully implement ranking yet because the database layer and richer data set are not in place. That adjustment keeps the enhancement practical while still leaving a clear path for more advanced database-backed query and ranking behavior later.

Overall, this milestone reinforced that algorithms and data structures are not limited to abstract programming exercises. They also shape how real applications retrieve, organize, and return data.
