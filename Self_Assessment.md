# Professional Self-Assessment

Completing the Computer Science program at Southern New Hampshire University has helped me connect my independent programming interests and my long-term professional goals. When I returned to school, part of my motivation was personal. I wanted to finish something important that I had started earlier in life. I also wanted to create new professional opportunities and build a path toward software development - a field that I am passionate about and have long held an interest in. As I moved through the program at SNHU, my goals became more focused, and I became more confident in my ability to learn new tools, solve technical problems, and build complete software systems.

One strength that I bring to this field is a strong interest in programming beyond required coursework. I have spent a significant amount of time developing in C and C++ on my own, including work with graphics programming and lower-level systems concepts. That experience has helped strengthen my fundamentals and has made it easier for me to understand new languages, frameworks, and architectural patterns. Even as the tools change, many core ideas remain connected: memory, data representation, control flow, abstraction, interfaces, debugging, and trade-offs in design. I still find programming enjoyable, and that sense of curiosity has been important to my growth throughout the Computer Science program.

This ePortfolio represents that growth through the enhancement of one primary artifact: the Travlr Getaways full-stack web application. The original version of Travlr Getaways was created in CS 465: Full-Stack Development I as a MEAN-stack application. For the CS 499 capstone, I enhanced the project across three major areas: software design and engineering, algorithms and data structures, and databases. The final version rebuilds the application as a React, Spring Boot, and PostgreSQL full-stack system with public trip browsing, database-backed search and filtering, protected administrative trip management, validation, security configuration, automated tests, and professional documentation.

One of the most important ways the program helped me grow was by improving how I approach software design and engineering. Earlier in the program, I often focused mainly on getting individual assignments to work. Over time, I began thinking more about structure, maintainability, separation of responsibilities, and how a project would be understood by another developer. The Travlr enhancement reflects that growth. The final application separates frontend components, API access code, backend controllers, DTOs, service logic, repository behavior, configuration, and database migration scripts. This structure makes the project easier to test, document, extend, and explain. It also reflects professional software development practices more clearly than the original mixed structure of static pages, server-rendered views, API routes, database models, and a separate Angular admin interface.

The capstone also helped me demonstrate stronger skills in algorithms and data structures. In the enhanced Travlr application, trip data is no longer treated as a simple list that is returned all at once. The backend supports search, filtering, sorting, pagination, lookup by stable trip code, and summary aggregation. These features required me to think carefully about how users might request data, how invalid or incomplete inputs should be handled, and where query behavior should live in the application architecture. The final implementation uses structured criteria objects, repository-level queries, service coordination, and automated tests to verify that the behavior works correctly. This helped me practice managing trade-offs between flexibility and performance.

The database enhancement was another major area of growth. The original Travlr project used Mongoose and MongoDB as part of a MEAN-stack application. In the enhanced version, I migrated the application to PostgreSQL-backed persistence using Spring Data JPA and Flyway. This required designing a relational trip table, applying database constraints, creating migration scripts, seeding data, and connecting the backend repository layer to the database. I also added a local development reset script so the trip catalog could be restored after testing administrative create, update, and delete operations. This work helped me better understand how database design supports application reliability. Unique trip codes, required fields, validation rules, repository queries, and repeatable development data all work together to protect the consistency of the application. The database layer also supports future growth by giving the application a clearer persistence model than development-focused data handling.

Security was also an important part of the final enhancement at every step. The enhanced application separates public browsing behavior from protected administrative behavior. Public users can view and search trips without authentication, but create, update, and delete operations are placed under protected admin endpoints. Spring Security is used to require an authenticated admin user for those operations. The application also includes backend validation, defensive service-layer checks, filename validation for trip image names, CORS limited to the configured frontend origin, and user-safe frontend error messages for admin API failures. Before final submission, I reviewed the frontend dependency audit results, researched the available fixes, applied the updates, and confirmed that the audit reported zero remaining vulnerabilities. I also verified that the frontend production build and backend test suite still passed after the security polish was complete. These choices show a security mindset because they address not only whether the application works, but also how it might fail, how it might be misused, and how protected resources should be separated from public functionality. With more time, I would continue this work by replacing the lightweight development Basic Auth flow with a token-based authentication and authorization design using JWTs.

Communication and documentation were also important parts of this capstone. The code review video required me to explain the original artifact, identify areas for improvement, and describe the planned enhancement direction before beginning the major development work. That process helped me think like a developer communicating with peers or future stakeholders. The enhancement narratives and ePortfolio required a different kind of communication. Instead of only describing code, I had to explain why the changes mattered, what skills they demonstrated, and how the work aligned with the course outcomes. This helped me practice adapting technical communication for different audiences and contexts.

Collaboration was also part of my growth throughout the program, even when assignments were completed individually. Courses involving the software development life cycle, team planning, testing, and code review helped me understand how software projects depend on communication, shared standards, and decision-making. In the capstone, I applied those ideas by using a code review process, writing contextual comments, organizing the project into clearer layers, and documenting the final application so another developer could understand how to run, test, and evaluate it. The final project is not only a working application; it is a project that is easier to review, discuss, and continue developing.

The three enhancements in this portfolio fit together as one cohesive full-stack improvement. The software design and engineering enhancement created the architectural foundation by restructuring the application into a React frontend and Spring Boot backend. The algorithms and data structures enhancement made the trip catalog more useful by adding structured search, filtering, sorting, pagination, and summary behavior. The database enhancement gave the application persistent relational storage with migrations, constraints, and repository-backed data access. The final project then brought these areas together through frontend polish, protected admin CRUD operations, validation, testing, documentation, and security review.

Overall, this ePortfolio shows my ability to take an existing project, evaluate its limitations, plan meaningful improvements, and implement those improvements using modern software development practices. It demonstrates growth in full-stack development, backend architecture, database design, algorithmic problem solving, testing, documentation, and security awareness. It also reflects the kind of developer I am working to become: someone with strong fundamentals, practical full-stack experience, and a genuine love of programming. As I move forward professionally, I want to keep building on this foundation by improving my skills in software engineering, backend development, database-driven applications, and secure full-stack systems.
