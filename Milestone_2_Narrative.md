# CS 499 Milestone Two Narrative

## Artifact Description

The artifact selected for this enhancement is the Travlr Getaways web application originally developed in CS 465: Full Stack Development. The original project was built through a staged migration process that introduced full-stack development concepts over time. It began with static public pages, added Express routing and Handlebars views, introduced a REST API with MongoDB/Mongoose data access, and eventually added authentication support and an Angular admin interface. This made the project valuable as a learning artifact because it showed how a web application can grow across multiple layers. However, it also left the final version with a mixed architecture, which made it a strong candidate for a ‘software design and engineering’ enhancement.

For CS 499, I selected this artifact as the foundation for a broader migration from the original mixed static/MEAN-style architecture toward a cleaner React, Spring Boot, and PostgreSQL stack. Milestone Two focuses on the first part of that migration: the software design and engineering enhancement. The current milestone does not complete the final database migration, but it establishes the architectural foundation that will allow the project to continue evolving through later enhancements.

## Enhancement Summary

The primary improvement made during this milestone was beginning the migration of the Travlr application into a clearer React and Spring Boot structure. On the backend, I created a Spring Boot API in the app-api portion of the project and organized it around controller, service, repository, model, DTO, and configuration layers. On the frontend, I created a React client in the app-client portion of the project using Vite and connected it to the new API. Together, these changes create a working vertical slice of the migrated application: the React client requests trip data from the Spring Boot API, and the API returns trip data through a layered backend structure.

This milestone does not complete every part of the final migration. The PostgreSQL database, full authentication and authorization flow, and full site navigation are planned for later work. However, the milestone establishes the structure needed for those later enhancements. The backend now exposes read-only trip endpoints for retrieving all trips, retrieving one trip by public trip code, sorting trips, and filtering trips by maximum price. The frontend displays those trips through reusable components and includes loading, error, and empty result states. I also began adapting the original Travlr visual identity by reusing the logo, background imagery, navigation styling, and trip images in the new React interface.

## Design Improvements, Learning, and Challenges

The most significant improvement is the separation of concerns. In the enhanced backend, the controller is responsible for HTTP request handling, the service layer is responsible for application-level operations, and the repository interface defines the data access boundary. This structure is important because the current repository uses temporary in-memory trip data, but the application is being designed so PostgreSQL persistence can replace that implementation later without requiring the controller or frontend to be rewritten. The React frontend follows the same idea by separating API communication from presentation logic. The tripApi.js file centralizes backend requests, while TripList and TripCard handle display concerns.

This part of the enhancement helped me learn that project migration requires more than converting one technology into another. I had to think ahead about how the new project should be structured, how easy it would be for another developer to understand, and how the architecture could grow in later milestones. For example, it would have been possible to add create, update, and delete endpoints immediately, but doing so before authentication and authorization are designed would create unnecessary security concerns. Instead, I kept the milestone focused on read-only endpoints while preparing a TripRequest DTO with validation rules for future client-provided data.

The data model was also improved as part of this forward-looking design. The original project used display-oriented strings for several important trip fields. In the enhanced version, the Trip model uses more appropriate typed fields, including LocalDate for the start date, BigDecimal for price, and a numeric duration field. These choices support sorting, filtering, validation, and future PostgreSQL mapping. The max price filter is a simple example of why this matters: price can now be handled as a numeric value rather than as formatted display text.

The largest challenge was making design decisions before every final layer of the application existed. Because the database enhancement has not been completed yet, I had to avoid designing only for the current in-memory implementation. I also had to consider security risks early, such as unsafe write endpoints, uncontrolled image paths, raw HTML rendering, and hardcoded configuration values. I addressed those concerns by delaying write endpoints, treating trip descriptions as plain text, validating image filenames, and moving CORS settings into application configuration. Through these choices I am practicing anticipating future architectural, security, and maintainability needs while keeping the milestone achievable.

## Course Outcome Coverage and Reflection

This enhancement made strong progress toward the software design and engineering outcome because it uses well-founded tools and architectural techniques to implement a more maintainable full-stack application. The project now demonstrates a layered Spring Boot backend, a component-based React frontend, REST API communication, configuration management, and API-boundary validation. It also supports the communication outcome because the code is organized into meaningful packages and components, and this narrative explains the design choices behind the enhancement.

The enhancement also shows progress toward the data structures and algorithms outcome, though more of that will be built out for Milestone Three. The trip data can now be sorted and filtered based on typed fields. In the current milestone, those operations are performed in memory because the PostgreSQL layer has not yet been implemented. However, the service and repository structure leaves a clear path for moving filtering and sorting into database queries later.

The process reinforced that migration involves more than changing technologies; it also requires architectural redesign. I had to think carefully about how the project should be structured, how another developer would understand it, how it could grow in later milestones, and which security concerns should be addressed early. The largest challenge was making forward-looking design decisions before every final layer existed. I addressed this by building a working read-only vertical slice, delaying write endpoints until authentication and authorization can be planned, using validation for future request data, and preserving a clean path from the current in-memory repository to PostgreSQL-backed persistence.
