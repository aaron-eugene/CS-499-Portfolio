# CS 499 Milestone Four Narrative

## Artifact Description

The artifact selected for this enhancement is the Travlr Getaways web application originally developed in CS 465: Full Stack Development. The original project was built as a MEAN-stack application using MongoDB, Express, Angular, and Node.js. It included customer-facing travel content, REST API functionality, database integration, authentication support, and an Angular administrative interface for managing trip data. Earlier in CS 499, I selected this artifact as the foundation for a broader migration toward a React, Spring Boot, and PostgreSQL architecture.

Milestone Four focuses on the database enhancement. In Milestone Two, I created the initial React and Spring Boot vertical slice and established the layered backend structure. In Milestone Three, I added structured trip search, filtering, sorting, pagination, direct lookup, and summary behavior while still using an in-memory repository. That work intentionally preserved a repository boundary so the temporary in-memory implementation could later be replaced with PostgreSQL-backed persistence. Milestone Four completes that next step by adding a real relational database layer to the enhanced Travlr application.

For this milestone, I replaced the temporary in-memory trip repository with PostgreSQL persistence using Spring Data JPA and Flyway. The database enhancement adds versioned migrations and a relational trips table with constraints and indexes, which makes the trip catalog more durable, repeatable, and structurally reliable.

The enhancement also moves more of the application’s data access behavior into the database layer. Search, filtering, sorting, pagination, and summary aggregation are now supported through the PostgreSQL-backed repository rather than being handled only as in-memory collection processing. This creates a stronger database implementation while preserving the controller, service, and repository structure created during the earlier milestones.

## Enhancement Summary

The main improvement completed during this milestone was replacing the default in-memory trip repository with a PostgreSQL-backed persistence layer. I added Spring Data JPA, the PostgreSQL driver, and Flyway to the Spring Boot API. I then configured the application so Flyway manages schema creation while Hibernate validates that the entity mapping matches the database. This is important because it keeps database structure under explicit version control instead of allowing the application framework to silently create or rewrite tables.

I converted the Trip model into a JPA entity and added an internal generated database id. At the same time, I preserved the trip code as the stable public identifier used by the API. This was also an important design decision. Using an internal id gives the database a clean primary key that can support future relationships, such as bookings or itinerary records. Keeping the trip code unique and public allows existing routes such as retrieving a trip by code to remain clear and user-friendly. The user or frontend does not need to know the internal database id.

I added Flyway migration files to create and seed the database. The first migration creates the trips table with appropriate PostgreSQL column types, constraints, and indexes. The table includes a unique constraint on trip code, check constraints for trip-code format, duration, price, non-blank text fields, and controlled image filename format. The second migration inserts the initial trip records that were previously stored in the in-memory repository. This improves maintainability because the database setup is repeatable. A new local database can be created, and Flyway can apply the schema and seed data in order.

The repository layer was also enhanced. I added a Spring Data JPA repository for low-level database access and a JpaTripRepository adapter that implements the application-level TripRepository interface. This keeps the service layer dependent on the project’s own repository abstraction instead of coupling it directly to Spring Data JPA. The previous InMemoryTripRepository is still available only through a memory profile, but the default application path now uses PostgreSQL.

I also moved the search, filtering, sorting, and pagination behavior into the database-backed repository. The TripSearchCriteria DTO still represents optional query parameters from the API, but the PostgreSQL repository now builds database queries from those criteria. Keyword search checks visible catalog fields such as code, name, resort, and description. Numeric filters support price and duration ranges. Sorting is limited to approved fields so callers cannot sort by arbitrary entity properties. Pagination uses bounded page sizes to avoid unbounded result requests. These changes strengthen the database enhancement because PostgreSQL is now doing more than storing records; it is also supporting controlled data access patterns.

Finally, I moved catalog summary behavior into the repository layer. The TripSummary response still returns total trip count, distinct resorts, minimum and maximum price, and minimum and maximum duration. However, those values are now produced through database-backed aggregate queries rather than by scanning every trip in the service layer. This reduces unnecessary application-side work and shows more appropriate use of relational database capabilities.

## Design Improvements, Learning, and Challenges

This enhancement helped me better understand how database design affects the rest of an application. In the earlier milestones, the repository boundary seemed useful because it kept the controller and service layers separate from the temporary data source. In this milestone, that design decision paid off. The application could move from an in-memory repository to PostgreSQL without changing the controller endpoints or rewriting the main service flow. This reinforced the value of designing boundaries before every final implementation detail is in place.

I also learned more about the role of database migrations. Before this milestone, it would have been possible to let Hibernate create the database schema automatically. However, that would make the schema less intentional and harder to explain as a database artifact. Using Flyway made the database structure explicit. The migration files show exactly how the table is created, which constraints are enforced, which indexes exist, and how the initial data is inserted. This also helped me understand why versioned migrations are important for repeatable setup and future changes.

One important design challenge was deciding how to handle trip identity. The application already used trip code as a public identifier, and earlier milestones treated it as stable and unique. At the same time, a relational database often benefits from an internal generated primary key, especially when future tables may reference trips. I addressed this by using a generated id as the database primary key while enforcing trip code uniqueness with a database constraint. This keeps the API understandable while still giving the database a strong relational foundation.

Another challenge was deciding which logic belonged in the service layer and which logic belonged in the repository layer. In Milestone Three, search, filtering, sorting, pagination, and summary generation were implemented in the service because the project still used in-memory data. Once PostgreSQL was added, it made more sense for those operations to move closer to the database. The service now coordinates trip operations, while the repository handles database-backed query behavior and summary aggregation. This made the code cleaner, but it also required updating comments and tests so they reflected the new responsibilities.

Security and data integrity were also important considerations. I did not add public create, update, or delete endpoints during this milestone because write operations should be protected by authentication and authorization. The public API remains read-only. At the database level, the enhancement improves integrity through a dedicated application database user, a unique trip-code constraint, check constraints, controlled column lengths, and validation-aligned table design. The repository also uses approved sort fields and bounded pagination so user-controlled query parameters do not directly control arbitrary database behavior.
